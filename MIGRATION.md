# Dashboard migration plan — static builder → component app

## Why
The dashboard is generated by `dashboard-builder.js` (~10k lines) which emits a
single static `public/index.html` (~13k lines), deployed to Vercel. This works,
but it is the root cause of most of the issues we've hit: logic duplicated across
views, an orphaned panel that rendered but was unreachable, status/compliance math
inlined into markup, and "every change is surgery on one giant file."

Goal: replace **only the presentation layer** with a small component app, while
keeping the data pipeline that already works.

## What stays (do NOT touch)
- The Node/Playwright scraper (`index.js`, `scraper.js`, `platform-scrapers.js`,
  `license-scraper.js`) and its JSON outputs (`history.json`, `platform-data.json`,
  `costs.json`, `licenses.json`, `state-requirements.json`, …). This is the data
  layer and it's sound.
- `utils.js` — `getStatus`, `licenseStatus`, `computeComplianceSummary`, date and
  lookback helpers. These are now tested and become the **shared source of truth**
  imported by the frontend, so the math can't diverge again.
- `middleware.js` (the Vercel login gate), `vercel.json` headers, `robots.txt`.
  These are deploy-agnostic and carry over unchanged.
- Excel/PDF/email exporters — they read the same JSON, independent of the UI.

## Recommended stack
Vite + Preact (or React) with TypeScript, building to a static bundle. Rationale:
it's an internal dashboard reading pre-generated JSON (no server needed), so a
static SPA is the simplest fit and deploys to Vercel exactly like `public/` does
today. (A prior `nextjs-app/` existed and was retired — its component breakdown is
a useful reference, but Next.js's server features aren't needed here.)

## Phased cutover (each phase ships independently; static build stays the default until parity)
1. **Scaffold + data layer.** New `web/` app. A typed `loadData()` that reads the
   same JSON files as build inputs. Port `utils.js` (or import it) so status and
   compliance are computed once. Render the **Overview** (status cards → urgency →
   `computeComplianceSummary` scorecard → timeline). ~1–2 days.
2. **Team View.** The big one — port each sub-view as a component (All / Table /
   Priority / Kanban / By Deadline / By State / By Type / Pinned / AANP /
   Compliance / Timeline). Replace the index-based `showProviderView` toggle with
   real routing/state. ~3–5 days.
3. **Platforms, Reports, Help.** ~2–3 days.
4. **Parity pass + cutover.** Diff old vs new for every number; switch Vercel's
   build/output from `public/` to `web/dist`; keep `dashboard-builder.js` runnable
   as a fallback for one cycle.
5. **Delete.** Once parity is confirmed in production, remove `dashboard-builder.js`
   and `dashboard-helpers.js`. The scraper keeps emitting JSON; only the consumer
   changed.

## Risks / mitigations
- **Hidden behavior in the 13k-line HTML** (focus mode, localStorage prefs, print
  views). Mitigation: inventory every `localStorage` key and `onclick` before
  deleting; phase 4 is an explicit parity diff, not a guess.
- **Number drift during migration.** Mitigation: both old and new import the same
  `utils.js`, and the compliance tests already lock the metric.
- **Scope creep.** Keep feature parity first; redesign second.

## Rough effort
~2 focused weeks for parity, then the delete. Can be done incrementally without a
freeze — the static build stays live until `web/` reaches parity.

## Suggested first step
Scaffold `web/` (phase 1) as a proof-of-concept rendering the Overview from real
JSON. That de-risks the stack choice and the `utils.js` sharing before committing
to the larger Team View port.
